# Code Review 가이드 – Compose Multiplatform (KMP + CMP)

본 저장소는 **Compose Multiplatform(CMP)을 사용하는 Kotlin Multiplatform(KMP) 프로젝트**입니다.  
코드 리뷰의 목적은 스타일 통일이 아니라 **안정성, 성능, 유지보수성 확보**에 있습니다.

과도한 추상화나 교과서적인 개선 제안보다는,  
**실제 운영 환경에서 문제가 될 수 있는 지점**을 우선적으로 검토합니다.

---

## 1. 프로젝트 맥락

### 플랫폼 구성
- Kotlin Multiplatform (Android / iOS / Desktop)
- UI는 Compose Multiplatform 기반
- 가능한 경우 UI 로직은 공통 모듈에서 공유

### 설계 방향
- 플랫폼 의존성은 최소화
- 명시적이고 단순한 설계를 선호
- “잘 동작하는 코드”를 “과도하게 추상화된 코드”보다 우선

---

## 2. 메모리 누수 (Memory Leak) 방지

### 리뷰 중점
플랫폼을 넘나드는 환경에서 **메모리 누수 가능성**을 적극적으로 확인합니다.

### 검토 기준
- 장수 객체가 다음을 직접 참조하지 않는지 확인
    - Android Context
    - Activity / View / UIViewController 등 UI 객체
- `remember`, `DisposableEffect`를 사용한 리소스 정리 여부 확인
- Coroutine Scope가 명확한 생명주기를 가지는지 검토

### 권장 사항
- UI 생명주기에 종속된 객체는 반드시 해제 로직을 포함
- 플랫폼 객체를 저장해야 하는 경우 WeakReference 또는 명시적 해제 사용
- 전역 상태에서 UI 참조를 보관하지 않도록 주의

### 경고 신호
- Singleton 또는 static 객체가 UI나 Context를 참조
- 취소되지 않는 Coroutine Scope
- `DisposableEffect`에서 onDispose 누락

---

## 3. 불필요한 리컴포지션 & 성능

### 리뷰 중점
**불필요한 리컴포지션**으로 인한 성능 저하를 방지합니다.

### 상태 관리
- 상태 객체는 가능하면 불변(immutable)으로 설계
- `data class` 사용 시 `equals()` 동작이 리컴포지션에 미치는 영향 고려
- 안정적(stable)이지 않은 객체를 Composable 파라미터로 전달하지 않도록 주의

### Compose 설계 기준
- `remember`, `derivedStateOf`는 의도가 명확한 경우에만 사용
- Composable 내부에서 무거운 연산 수행 지양
- 역할별로 Composable을 분리하여 리컴포지션 범위 최소화

### 경고 신호
- mutable 객체를 그대로 Composable에 전달
- Composable 내부에서 객체 / 람다를 매번 새로 생성
- `remember`로 문제를 감추는 구조

---

## 4. 크래시 대응 및 안정성

### 리뷰 중점
크래시는 **사전에 방지**하고, 발생 시 **원인 파악이 가능하도록** 설계되어야 합니다.

### 검토 기준
- null, 캐스팅, 인덱스 접근에 대한 방어 코드 존재 여부
- 플랫폼별 크래시 가능 코드가 적절히 분리되어 있는지 확인
- 공통 모듈에서 플랫폼 API를 직접 사용하지 않는지 검토

### 에러 처리
- 예외 발생 시 로그를 남기되, 무조건 삼키지 않기
- 치명적이지 않은 경우 graceful degradation 우선
- 실패 가능성이 있는 코드의 흐름이 명확한지 확인

### 경고 신호
- 근거 없는 `!!` 사용
- catch 후 아무 처리도 하지 않는 코드
- 공통 코드에서 플랫폼 전용 API 호출

---

## 5. 객체지향 설계 & 책임 분리

### 리뷰 중점
객체의 **책임이 명확하고 응집도가 유지되는지**를 확인합니다.

### 설계 원칙
- 클래스는 하나의 명확한 책임만 가질 것
- 상속보다는 조합(Composition) 우선
- UI, 상태, 비즈니스 로직의 역할 분리

### Compose 관점
- Composable은 “그리기”에 집중
- 비즈니스 로직이나 플랫폼 제어는 외부로 분리
- Side-effect는 명시적이고 제한된 범위에서만 사용

### 경고 신호
- 하나의 클래스가 여러 역할을 수행
- 깊은 상속 구조
- UI 컴포넌트가 로직과 플랫폼 제어를 직접 수행

---

## 6. 멀티플랫폼 공통 코드 기준

### 공통 모듈 규칙
- Android / iOS 플랫폼 API 직접 사용 금지
- 플랫폼 차이는 interface 또는 expect / actual로 격리
- 공통 코드는 테스트 가능하고 결정론적이어야 함

### expect / actual
- 플랫폼 차이가 불가피한 경우에만 사용
- 공통 인터페이스에 플랫폼 개념이 노출되지 않도록 주의

### 우선순위
1. 안정성
2. 성능
3. 유지보수성
4. 코드 의도 명확성

### 지양할 사항

- 불필요한 아키텍처 제안
- 앱 전용 패턴을 SDK / 공통 UI 코드에 강요

- 멀티플랫폼 환경에서 안전한가?
- 불필요한 리컴포지션을 유발하지 않는가?
- 크래시 가능성을 충분히 고려했는가?
- 책임이 명확하고 장기 유지보수가 가능한가?
